Modules

Les fonctionnalités du framework sont découpées en modules. L'idée de ces modules est de fournir à l'éditeur des fonctionnalités complexes d'un point de vue scripting qu'il peut configurer très simplement depuis un fichier de configuration. Ainsi, chaque module possède à sa racine un fichier nommé config.hpp qui permet au minimum de déactiver totalement le module. Exemple du fichier de configuration du module actionsFlags qui permet de créer des drapeaux utilisables par les autres modules pour l'ajout d'actions par exemple :

modules/actionsFlags/config.hpp

Nommage des fichiers

Si vous souhaitez créer votre module, merci d'observer quelques règles de nommage afin de conserver l'homogénéité et la logique. Ces règles devraient pouvoir être respectées dans la majorité des cas.

Un module est composé au minimum d'un fichier config.hpp et d'un dossier content. Tout ce qui n'est pas sensé être visible pour l'utilisateur final (scripts, fonctions, medias...) devrait aller dans le dossier content. Dans la grande majorité des cas, votre module prendra cette forme config.hpp + content. Le seul qui pour le moment ne respecte pas cette règle est le module d'intro qui contient un script qui doit être édité par l'éditeur, donc à la racine du module.

Les scripts déclenchés automatiquement, au démarrage de la mission se nomment initScript.sqf. Si un script est spécifique au serveur ou au client on le sufixe : initScriptClient.sqf, initScriptServer.sqf. Les scripts d'initialisation sont inclus dans core/defaultClient.sqf et core/defaultServer.sqf.

Les scripts d'interface sont nommés uiScript.sqf, on trouve généralement un script uiScriptStart.sqf qui initialise toute la logique de l'interface lorsque celle-ci est ouverte et un script uiScriptStop.sqf dans lequel on détruit les variables utilisées par la logique d'interface et où l'on supprime tout ce qui n'est plus nécessaire. La config de l'interface se trouve dans un fichier nommé ui.hpp et est incluse directement dans le description.ext. les IDD et IDC se trouvent dans un fichier nommé ctrl.hpp et est inclus dans ui.hpp.

Les scripts déclanchés par une action du joueur comme la validation d'un saut en parachute depuis l'interface du module paraJumpAdvanced par exemple, se nomment script.sqf, scriptClient.sqf et scriptServer.sqf.

Le module paraJumpAdvanced est un bon exemple de ces différentes règles de nommage.
